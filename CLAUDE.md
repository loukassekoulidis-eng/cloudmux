## Architecture

cloudmux is a cloud identity multiplexer. It creates isolated config directories per cloud profile under `~/.cloudmux/profiles/<name>/` and sets provider-specific env vars (`AZURE_CONFIG_DIR`, `CLOUDSDK_CONFIG`, `AWS_CONFIG_FILE`, etc.) so each terminal can use a different cloud identity simultaneously. It never wraps or proxies cloud CLI commands — it only manages environment isolation.

### Key data flow

`cloudmux use <profile>` → load profile YAML → resolve provider plugin → provider returns env var map → validate directory permissions → shell hook exports env vars into current shell.

`cloudmux login <profile>` → same profile resolution → create isolated dirs (0700) → provider executes native CLI login (e.g., `az login --tenant X`) with isolated config dir → native CLI caches tokens there.

### Layer separation

- **`cmd/cloudmux/main.go`** — Entry point only
- **`internal/cli/`** — Cobra command definitions. Thin layer: parse args, call session manager, format output
- **`internal/session/`** — Core orchestrator. Connects profiles, providers, env isolation, and health checking
- **`internal/config/`** — YAML loading/validation for `config.yaml` and `profiles.yaml`
- **`internal/provider/`** — Provider interface + implementations (azure/, gcp/, aws/, custom/). Each provider knows its env vars, login/logout/status commands
- **`internal/security/`** — Permission enforcement (0700 dirs, 0600 files), profile name sanitization (`[a-zA-Z0-9_-]`, max 64 chars), audit logging
- **`internal/shell/`** — Shell hook generation (bash/zsh/fish templates). The hook defines a shell function that wraps the binary for commands needing to modify the shell environment (`use`, `logout`)

### Provider interface

All providers implement `provider.Provider`: `Name()`, `EnvVars(profile)`, `Login(profile)`, `Logout(profile)`, `Status(profile)`, `Validate(profile)`. Registry maps provider names to implementations.

### Security invariants

- All profile directories: `0700`. All files: `0600`. Enforced on every operation that touches the filesystem.
- Profile names: `[a-zA-Z0-9_-]`, max 64 chars, no leading `-` or `.`, no reserved names. Validated at load time, not just creation.
- No shell expansion on config values. Only `{profile_dir}`, `{name}`, `{home}` template vars. Custom provider commands are the sole exception (user-authored, documented).
- Native CLI commands executed via `exec.Command` with explicit args, never `sh -c` (except custom provider).

## Implementation Phases

Phase 1 (MVP): Go scaffolding, profile system, Azure provider, shell hooks (bash/zsh), core commands, permission enforcement.
Phase 2: GCP + AWS + custom providers, doctor command, fish support, shell completions.
Phase 3: gc command, TTL, audit logging, confirm_on_use, colored output.
Phase 4: goreleaser, Homebrew, CI.

## Git & Authorship

This project must look entirely human-authored. No AI fingerprints anywhere:
- **No `Co-Authored-By` trailers** in commits. Ever.
- Commit messages should be natural and terse — the way a developer actually writes them (`add profile validation`, `fix permission check on login`, `wire up azure provider`). No corporate-speak, no bullet lists in commit bodies.
- Commit in logical, human-sized chunks. One concern per commit. Don't dump an entire feature in a single commit, and don't make trivially small commits either.
- No "Generated by" or "Created with" comments in code or docs.
- No comments explaining obvious code. Comments only where a human would leave one.
- PR descriptions (when applicable) should be casual and brief, not templated.
